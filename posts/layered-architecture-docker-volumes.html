Layered Architecture & Docker Volumes

How Docker's layered image model works, why container data is ephemeral by default, and how volumes solve the persistence problem.

What is Layered Architecture?

Docker images are built in layers — each layer represents a single instruction in the Dockerfile. These layers are essentially a set of filesystem changes stacked on top of one another.

Advantages

  • Reusability: Docker reuses layers across builds if their contents and instruction hashes are identical, making builds faster and lighter.
  • Caching: Layer caching dynamically boosts performance and storage efficiency, especially in CI/CD workflows.

Copy-on-Write Mechanism

A container generally has two layers:

  1. Image Layer — Contains the original layers of the image. Immutable.
  2. Container Layer — Contains files that are modified or created at runtime. These are ephemeral — this data is destroyed as soon as the container is removed.

When you create new files or modify existing files from the image layer, a copy is created in the container layer. This keeps the original image intact. That mechanism is called Copy-on-Write.

Why Do We Need Docker Volumes?

Since the data in the container layer is ephemeral, volumes are the solution for retaining data even after a container is stopped, removed, or recreated.

Advantages of Volumes

  1. Persistence across restarts — Even if the container dies, the data remains intact if a volume is attached.
  2. Data sharing between containers — Multiple containers can read and write to the same volume simultaneously.
  3. Decoupling data from images — Volumes keep persistent data separate from the image, keeping images clean and reusable.

Types of Volumes in Docker

1. Named Volumes

Stored in Docker's internal directory at /var/lib/docker/volumes and fully managed by Docker. 

# Creating a named docker volume
docker volume create volume_name

# Attaching the volume while running the container
docker run -v volume_name:/path/on/container image

2. Bind Mounts

Mounts a specific path on the host system directly into the container. You control the host path. 

# Attaching a host path while running the container
docker run -v /path/on/host:/path/on/container image

3. Anonymous Volumes

Similar to named volumes but without a user-defined name. Docker assigns a random identifier automatically. 

# Attaching a volume anonymously to a container
docker run -v /path/on/container image

Volume Commands Cheat Sheet

# Create a named volume
docker volume create volume_name

# List all Docker volumes
docker volume ls

# Inspect a specific volume
docker volume inspect volume_name

# Remove a volume
docker volume rm my_volume

# Remove all unused volumes
docker volume prune

# Attach a named volume while running a container
docker run -v my_volume:/data my_image

# Run a container using --mount syntax
docker run --mount source=my_volume,target=/data my_image

# Run with a bind mount from host path to container path
docker run -v /host/path:/container/path my_image

# Run a container with an anonymous volume
docker run -v /data my_image
💡 Prefer named volumes over anonymous volumes in production — they're easier to inspect, back up, and migrate between environments.
